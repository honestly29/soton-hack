❯ wait what does repomix provide then?                                          

⏺ Repomix takes a repo and produces a single LLM-ready document containing:

  1. Directory tree — the full file structure
  2. File contents — either full text or Tree-sitter compressed (signatures
  only, no function bodies)
  3. Token count — per-file and total
  4. Security scan — flags accidentally committed secrets
  5. Git history — recent commits, optionally diffs

  Output formats: XML, Markdown, JSON, or plain text.

  So for our use case: user pastes github.com/owner/repo → we call Repomix as a
  library → it clones, extracts, compresses → we get a single string of the
  repo's product context (README, docs, architecture skeleton) ready to feed to
  the ingestion agent.

  It replaces what we'd otherwise build manually: cloning, walking the file
  tree, filtering relevant files, extracting text, managing token budgets. It's
  the "repo → LLM context" step handled for us.

⏺ Really good research. Here's the key finding:

  The industry consensus is hybrid, not either/or

  Pure context stuffing (Repomix-style full dump) has real problems:
  - "Context rot" — irrelevant context actively harms reasoning, it's not
  neutral padding
  - "Lost in the middle" — 30%+ accuracy drop for info buried in large contexts
  - A medium repo can be 500k-2M tokens — way beyond useful

  Pure agentic exploration (give tools, let it wander) is overkill for product
  understanding:
  - Designed for code editing / bug fixing, not "what does this product do"
  - Multi-turn exploration adds latency and cost
  - Agent can miss files it doesn't know to look for

  What every serious tool actually does (Claude Code, Cursor, Aider, Copilot):
  1. Upfront skeleton — lightweight structure map (directory tree, signatures,
  README)
  2. On-demand drill-down — tools to read specific files when needed
  3. Summarize — compress findings before reasoning

  For our use case (product understanding)

  The sweet spot is "targeted stuffing" — not "stuff everything" but "stuff the
  right things":

  1. Use Repomix to clone + extract only README, docs/, package.json, directory
  tree, route definitions, data models (~5-20 files, 5k-50k tokens)
  2. Use Tree-sitter compression for the rest (signatures only, no function
  bodies)
  3. Single generateObject call with curated context →
  ProductProblemRepresentation

  This is Repomix used smartly with include filters, not a blind full dump. A
  curated 20k-token context dramatically outperforms either a 500k-token dump or
   a 10-round agentic exploration for this task.

  Does this approach make sense? Or do you want the agent to have tools to drill
   into files if the initial extraction is thin?
